# コンテンツ・スタイルルール

スライドコンテンツの品質と一貫性のためのガイドライン。

## コンテンツルール

### 1. 1スライド1メッセージ

- 各スライドはひとつの主張だけを伝える
- 冗長な情報は削除する
- 網羅性より明瞭さを優先する

### 2. スライド単体で完結させる

- どのスライドも単体で理解できるようにする
- 「前のスライドで述べたように」「先ほどの〜」といった参照は使わない
- 必要な文脈はそのスライド内に含める

### 3. プレースホルダー禁止

- すべてのテキストは具体的に記述する
- 「[ここにデータを挿入]」や「未定」は使わない
- ソース素材をもとに実際のコンテンツを生成する

### 4. 広さより深さ

**表面的な箇条書きではなく、具体的で説得力のある内容を書く。**

| NG（薄い内容） | OK（深い内容） |
|---|---|
| `- パフォーマンスが向上する` | `- レスポンス時間が平均200ms→50msに短縮、ユーザー離脱率が15%改善` |
| `- コスト削減` | `- 年間インフラコストを40%削減（月額$12K→$7.2K）、ROIは6ヶ月で回収` |
| `- 使いやすい` | `- 初回セットアップ5分、設定ファイル不要で既存CIに即統合` |
| `- セキュリティ対策` | `- SOC2 Type II準拠、E2E暗号化、90日ごとの第三者監査を実施` |

各箇条書きには以下のうち少なくとも1つを含める:
- **具体的な数値**（%、時間、金額、件数）
- **比較**（Before/After、従来手法との差）
- **具体例**（実際のユースケース、ツール名、手順）
- **因果関係**（なぜそうなるか、どういう仕組みか）

### 5. 聴衆の関心を引く

- **問いかけ**: 「〜という経験はないでしょうか？」で始めると共感を得やすい
- **意外な事実**: 通説と異なるデータや事実を提示する
- **ストーリー**: 抽象論より、具体的なシナリオで語る
- **対比**: Before/After、従来/新方式の対比で変化を際立たせる

## スタイルルール

### 1. 物語型の見出し

見出しはラベルではなく、ストーリーを語る。

| NG | OK |
|-----|------|
| "主要な統計" | "利用者数が6ヶ月で2倍に" |
| "ソリューション" | "1つのプラットフォームが5つのツールを置き換える" |
| "メリット" | "チームは週10時間を節約できる" |
| "概要" | "成長を支える3つの柱" |
| "背景" | "10年のイノベーションがここに集約" |
| "課題" | "開発者の40%が週5時間をデプロイ作業に浪費" |
| "機能紹介" | "1コマンドで本番環境に安全にデプロイ" |

### 2. AI的な定型表現を避ける

以下のパターンは削除する:
- 「〜を探求する」「〜の旅」「〜のランドスケープ」
- 「見てみましょう」「ご紹介します」
- 「革新的な」「画期的な」「ゲームチェンジャー」
- 「まとめると」「要約すると」「結論として」
- 「活用する」「レバレッジする」「エンパワーする」

### 3. 意味のあるクロージング

「ありがとうございました」「質問は？」だけで終わらない。

以下のいずれかを含める:
- 具体的な次のステップを示すコールトゥアクション
- データに裏付けられた印象的な要点
- 考えさせられる締めの一言
- 具体的なタイムラインやURLを含む次のステップ

### 4. Markdownフォーマット

**タイトル**: カバーは `#`、スライドタイトルは `##`、セクション小見出しは `###`。

**本文**:
- 箇条書きには `-` を使用（`*` は使わない）
- 重要な用語や数値は `**太字**` で強調
- 引用には `>` を使用
- 各箇条書きは具体的な詳細を含む2〜3行（単なるフレーズではない）
- 1スライドあたり最大5項目

**テキスト階層**:
- 1つのテキスト要素に `#` または `##` 見出しは1つだけ
- 本文はタイトルとは別の要素にする
- 1つの要素内で見出しレベルを混在させない

## プレゼンテーションの流れ

### 冒頭（スライド1〜2）

| スライド | 目的 |
|---------|------|
| カバー | タイトル + フック + ビジュアルアイデンティティ |
| 概要 | アジェンダまたは問題提起 |

### 中盤（コンテンツスライド）

**構成パターン**:

| パターン | 使い所 |
|---------|--------|
| 課題 → 解決策 | 新しいプロダクト・アイデアの紹介 |
| 何を → なぜ → どうやって | 教育的コンテンツ |
| 過去 → 現在 → 未来 | 変革のストーリー |
| 主張 → 根拠 → アクション | データドリブンな議論 |

### 締め（最終スライド）

| 要素 | 目的 |
|------|------|
| 要点 | 核心メッセージの強化 |
| コールトゥアクション | 明確な次のステップ |
| 印象的な締め | 心に残る一言 |

## コンテンツ密度

| レベル | ガイドライン |
|--------|-------------|
| タイトルスライド | タイトル + サブタイトルのみ |
| コンテンツスライド | タイトル + 3〜5項目の箇条書き（各2〜3行で詳細を含む） |
| 引用スライド | 引用 + 出典のみ |
| 統計スライド | 数値1つ + 文脈 |
| セクション区切り | セクション名のみ |

## 言語処理

**検出の優先順位**:
1. `--lang` フラグ（明示指定）
2. ユーザーの会話言語
3. ソースコンテンツの言語

**ルール**: 検出した言語ですべてのコンテンツを生成する:
- スライドのテキスト
- 進捗レポートや確認メッセージ
- エラーメッセージ

技術的な用語（ファイルパス、JSONキー）は英語のまま。
