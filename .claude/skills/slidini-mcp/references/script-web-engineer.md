# WEBエンジニア向け台本ガイド

## ターゲット視聴者

WEBエンジニア（初級〜中級）を主なターゲットとする。ただし、ユーザーが別のレベルを指定した場合はそちらに合わせる。技術的に正確であることが最優先だが、専門用語を使う際は初出時に簡潔な補足を入れることで、幅広い層にリーチできるようにする。

## トーン

「です・ます」調の丁寧でプロフェッショナルな語り口を基本とする。堅すぎず、かといって砕けすぎず、「信頼できる先輩エンジニアが解説している」ような雰囲気を目指す。

## フックの切り口

技術系の視聴者は「自分の業務に直結する話」に強く反応する。以下のようなパターンが効果的：

- **課題提起型**: 「こんな経験ありませんか？」「○○で困ったことは？」
- **意外性型**: 「実は○○は□□だった」「知らないと損する○○」
- **速報型**: 「○○がついにリリースされました」「○○に大きなアップデートが」
- **比較型**: 「○○と△△、結局どっちがいいの？」

## 本題の構成パターン

技術トピックでは以下の構成が効果的：

- **概念説明 → コード例 → 実践的なユースケース**: 新技術の紹介に最適
- **Before/After → 仕組み解説 → 導入方法**: 既存技術の改善・移行ガイドに最適
- **比較軸の提示 → 各技術の深掘り → 選定基準**: 技術比較・選定ガイドに最適

## ビジュアル要素

共通のビジュアル要素に加え、以下を積極的に活用する：

### コードブロック

技術トピックではコード例が非常に効果的。スライドに表示するコードは短く保ち（10行以内が目安）、重要な行にはコメントで注釈を付ける。

```javascript
// Bad: コールバック地獄
getData(function(a) {
  getMoreData(a, function(b) {
    getEvenMoreData(b, function(c) {
      console.log(c);
    });
  });
});
```

### アーキテクチャ図

システム構成やデータフローを説明する際は、Mermaid記法でフローチャートやシーケンス図を指示する。

### ターミナル出力

CLIツールやコマンドの実行結果を見せることで、実践的な雰囲気を演出できる。

## まとめでの行動喚起

- 公式ドキュメントを読む
- 実際にコードを書いて試す
- GitHubリポジトリをチェックする
- 関連動画を見る

## ファクトチェックの重点項目

技術系コンテンツでは「なんとなく正しそう」が最も危険。視聴者はコード例やバージョン番号の誤りにすぐ気づく。特に以下を重点的に検証する：

- バージョン番号・リリース日（公式リリースノートやGitHubタグで確認）
- API名・設定ファイルのパス・ディレクトリ構造（公式ドキュメントで確認）
- コード例の構文（非推奨APIを使っていないか）
- CLIコマンド・オプション（man pageやヘルプで確認）
- パフォーマンス数値の出典（ベンチマークの条件を揃えて比較しているか）

## 留意事項

- コード例は必ず動作確認済み（または動作が自明）なものを使う
- 更新頻度の高いツール（フレームワーク、CLIツール、AI関連サービスなど）は特に注意。公式ドキュメントの最新版で確認すること
- 技術の良し悪しを断定しすぎない。トレードオフを公平に提示する
